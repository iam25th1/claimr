Auth fix v2 - widen "already initialized" catch

What's in your error log: "Could not start PIN challenge: The user had
already been initialized." That's Circle rejecting the createUserPinWithWallets
call because the user's PIN and wallet are both already set up. My
previous catch only checked numeric error codes (155206, 155709) and
Circle returned a different code for this case.

This patch broadens the detection to also match the error message text,
which is more reliable across Circle's various endpoint variants.

Patterns now caught:
  - code 155206 (pinAlreadySet)
  - code 155207 (userInitialized)
  - code 155709 (walletAlreadyExists)
  - any message containing "already initialized" / "already been initialized"
  - any message containing "already set" / "already exists"
  - any message containing "wallet already"

When isAlreadyInit fires, the route returns alreadyInitialized=true and
the client (providers.tsx, already shipped in the previous batch) skips
the PIN sheet and goes straight to finalize, which looks up the existing
wallet for that email and sets the session cookie.

Net effect: returning users sign in without seeing this error. The first-
time PIN sheet still fires for genuinely new users.
