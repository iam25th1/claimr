MetaMask connect + PIN-reduction reality check

What this batch does

  Adds MetaMask (and any injected wallet - Rabby, Brave, Coinbase Wallet
  extension, etc.) as an alternative sign-in option. Users now have two
  ways to authenticate:

    EMAIL FLOW (existing, unchanged)
      Sign up with email -> Circle creates a custodial-ish wallet ->
      Set a PIN -> PIN per transaction. Strong security, no extensions
      required, friendly to non-crypto users. But yes, lots of PIN entry.

    WALLET FLOW (new)
      Click "Connect MetaMask" -> MetaMask popup -> Approve -> Done.
      One click to sign each transaction. No PIN. No email. No Circle
      setup. Use your existing wallet.

The PIN question, honestly

The PIN-per-transaction is Circle's design. It's how user-controlled
wallets work: every signature requires the user to prove they're the
one authorizing. There's no setting in Circle's SDK that bypasses this
without giving up the security model.

What this batch DOES for the PIN problem:
  Users who switch to MetaMask flow stop having PIN entries entirely.
  MetaMask shows its own confirmation popup per transaction (typically
  one click each), which is meaningfully faster than the PIN sheet.

What this batch does NOT do:
  Circle users still face PIN-per-tx. For a multi-slot post with 5
  slots, that's still 1 approve + 5 postJobs = 6 PIN entries.

Genuine paths to reduce PIN further for Circle users (NOT in this batch):
  A.  Approve-max pattern. On the first swap or first post, approve
      uint256.max instead of the exact amount. Subsequent transactions
      skip the approve step entirely. Saves 1 PIN per repeat operation.
      Easy to add as a follow-up.
  B.  Contract-level batching. Add a `postJobs(...)` function to the
      Claimr escrow contract that takes an array and posts N jobs in
      one transaction. Drops slot posting from N+1 to 2 PIN entries
      (1 approve + 1 batch). Requires re-deploying the contract.
  C.  Circle session features. If Circle ever adds a "remember PIN for
      session" mode, we plug into it. Currently no such mode exists in
      the W3S SDK.

Test after deploy

  WALLET FLOW
  1. Open in incognito with MetaMask installed
  2. Click Launch App -> /dashboard/discover
  3. Click any Claim button -> bounced to /onboarding
  4. Scroll past email field, click Connect MetaMask
  5. Approve in MetaMask popup
  6. You land on /dashboard/discover, signed in as wallet user
  7. Click Claim on a job -> MetaMask popup pops (no PIN sheet)
  8. Confirm tx -> done, single click each

  EMAIL FLOW (unchanged)
  Existing flow still works identically for Circle users.

  COEXISTENCE
  If a user signs in with email AND happens to have MetaMask connected
  in the same browser, the Circle session takes precedence. Logging
  out clears both.


Notes for admin features

  Admin gates are still email-based (ADMIN_EMAILS env). MetaMask users
  cannot reach admin pages even if their wallet is in
  NEXT_PUBLIC_ADMIN_WALLETS, because the server-side check reads the
  Circle session cookie's email field. If you want wallet-based admin,
  that's a SIWE (Sign-In with Ethereum) integration - separate batch.
